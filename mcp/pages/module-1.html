Lesson 1-1: The Protocol and Three Primitives

25 min · Concept

The Model Context Protocol (MCP) is an open standard that lets Claude communicate with external servers using JSON-RPC 2.0. Think of it as a plug standard — any tool that speaks MCP can connect to any MCP-compatible AI client, including Claude Code.

How the Protocol Works

MCP defines a client-server architecture. Claude Code is the client; your server is the bridge to your tools.

text 
Claude Code (MCP client)
      ↕ JSON-RPC 2.0 over stdio or HTTP/SSE
MCP Server (your bridge)
      ↕ native SDK / REST / DB driver
Your System (Jira, internal API, database...)

Two transport options:

  • stdio — Claude spawns your server as a subprocess. Fast, local, no network needed. Best for local tools and development.
  • HTTP/SSE — Your server runs as a persistent service. Best for remote tools, shared team servers, and hosted integrations.

The Three Primitives

Every MCP server exposes some combination of these three building blocks:

text 
┌─────────────────────────────────────────────────────────┐
│  Tools       — Callable functions Claude can invoke     │
│               e.g. search-tickets, create-issue,        │
│               query-db, post-comment                    │
├─────────────────────────────────────────────────────────┤
│  Resources   — Data Claude can read (like files)        │
│               e.g. ticket://LIN-1234, schema://users,   │
│               metrics://api/p95-latency                 │
├─────────────────────────────────────────────────────────┤
│  Prompts     — Reusable prompt templates from server    │
│               e.g. "summarize-sprint", "review-PR"      │
└─────────────────────────────────────────────────────────┘

Tools vs Resources — When to Use Which

  • Use a Tool when Claude needs to do something — query, create, update, delete, search.
  • Use a Resource when Claude needs to read something specific by URI — like a file, a record, a schema.
  • Use a Prompt when you want the server to supply a reusable prompt template that changes based on server-side context.

Most real-world servers primarily expose Tools. Resources and Prompts are less common but powerful for specific use cases.

💡
Tools Are the Core Primitive
In practice, most MCP servers you build will expose only Tools. Start there. Add Resources when you have clearly addressable data (records with stable URIs). Add Prompts only if your server has context that should drive the prompt itself.

Lesson 1-2: MCP vs Skills vs Hooks — The Decision Framework

20 min · Concept

In the Claude for Software Engineers course you already have skills, subagents, and hooks. MCP is a fourth layer — and it's the most expensive to set up. Here's exactly when each approach wins.

The Honest Comparison

⚠️
Claude Can Already Fetch Public URLs
If your tool is publicly accessible — GitHub, Linear, Sentry — Claude can fetch it via URL in an interactive session. MCP adds overhead without payoff for public tools used interactively. Be honest with yourself about whether you actually need it before building a server.

Decision Table

Scenario Best Approach Why
Reusable task template (code review, commit message) Skill No external system needed — just instructions and command injection
Block a dangerous command, auto-lint after edit Hook Deterministic enforcement at a lifecycle event
Read a public GitHub issue or Linear ticket interactively URL fetch (no MCP) Claude can already do this — no server needed
Read from a private internal API or database MCP Claude cannot reach private systems without a bridge
Headless workflow that needs ticket data (no human present) MCP No human to paste content; needs programmatic access
Write back to a system — move ticket, post comment, update DB MCP URL fetching is read-only; Tools can take actions
Structured data from an API (not HTML scraping) MCP Tools return clean JSON; fetching returns HTML with noise

The Four Genuine MCP Advantages

  1. Private systems — Internal APIs, private databases, Confluence, internal Jira. Claude has no path there without your server as a bridge.
  2. Headless automation — In cron jobs and CI pipelines running claude -p, there's no human to paste content. MCP gives those workflows programmatic access.
  3. Write-back actions — Moving a ticket to "In Review", posting a PR comment, updating a database record. URL fetching is read-only. MCP Tools can mutate state.
  4. Structured data — A Tool returns the exact JSON fields you want. Fetching a URL returns raw HTML that Claude has to parse, which introduces noise and failures.
✅
Checkpoint
Before moving on, identify one real system in your workflow that hits at least one of the four advantages above — ideally a private internal tool or a write-back use case. That's what you'll connect in Module 4.
