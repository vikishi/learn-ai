Private Systems & Write-Back

Lesson 4-1: Connecting Private Internal Systems

25 min · Hands-on

This is the scenario where MCP has no substitute — private systems that Claude fundamentally cannot reach by fetching URLs. Here's how to connect them safely.

Pattern 1: Internal REST API

A tool that calls your internal API with a service account token:

typescript 
// Auth: read token from env, never hardcode
const INTERNAL_API_TOKEN = process.env.INTERNAL_API_TOKEN;
const INTERNAL_API_BASE = process.env.INTERNAL_API_BASE ?? "https://api.internal.mycompany.com";

server.tool(
  "get-feature-flags",
  "Get current feature flag values for a service or user segment.",
  {
    service: z.string().describe("Service name, e.g. 'checkout', 'auth'"),
    environment: z.enum(["staging", "production"]).default("staging"),
  },
  async ({ service, environment }) => {
    const res = await fetch(`${INTERNAL_API_BASE}/flags/${service}?env=${environment}`, {
      headers: { Authorization: `Bearer ${INTERNAL_API_TOKEN}` },
    });
    if (!res.ok) throw new Error(`API error: ${res.status}`);
    const flags = await res.json();
    return { content: [{ type: "text", text: JSON.stringify(flags, null, 2) }] };
  }
);

Pattern 2: Internal Database

Query a private database — be explicit about read-only access in the tool description to set correct expectations with Claude:

typescript 
import postgres from "postgres";

const sql = postgres(process.env.DATABASE_URL!);

server.tool(
  "query-metrics",
  "Run a read-only query against the metrics database. Use for answering questions about usage, errors, and performance. Do not use for writes.",
  {
    query: z.string().describe("SQL SELECT query only"),
  },
  async ({ query }) => {
    // Guard: reject anything that isn't a SELECT
    if (!/^\s*SELECT/i.test(query)) {
      throw new Error("Only SELECT queries are allowed");
    }
    const rows = await sql.unsafe(query);
    return { content: [{ type: "text", text: JSON.stringify(rows, null, 2) }] };
  }
);

Auth Patterns

Pattern How Best For
API Key -e API_KEY=xxx in claude mcp add Simple internal APIs
Service Account Dedicated non-personal credentials in env Production systems, team servers
Shell Profile Export in ~/.zshrc or ~/.bashrc Developer laptops, personal access
Secret Manager Load from 1Password CLI, AWS Secrets Manager at shell start Regulated environments, shared CI
🛑
Never Commit Credentials
Tokens in .mcp.json get committed and shared. Always use ${ENV_VAR} interpolation in .mcp.json and store the actual values in your shell profile or a secret manager — never in the file itself.

Lesson 4-2: Write-Back Actions and Team Deployment

25 min · Hands-on

Read access is useful. Write-back is where MCP completes the loop — Claude can now take actions on your systems, not just read from them.

Example: Jira Write-Back Tools

Three tools that together enable the full ticket-to-PR workflow:

typescript 
const JIRA_BASE = process.env.JIRA_BASE!;
const JIRA_TOKEN = process.env.JIRA_TOKEN!;

const jira = (path: string, method = "GET", body?: object) =>
  fetch(`${JIRA_BASE}/rest/api/3${path}`, {
    method,
    headers: {
      Authorization: `Basic ${Buffer.from(process.env.JIRA_EMAIL + ":" + JIRA_TOKEN).toString("base64")}`,
      "Content-Type": "application/json",
    },
    body: body ? JSON.stringify(body) : undefined,
  }).then((r) => r.json());

// Read: get ticket details
server.tool(
  "get-ticket",
  "Fetch details of a Jira ticket by ID, including description, acceptance criteria, and current status.",
  { id: z.string().describe("Ticket ID, e.g. ENG-1234") },
  async ({ id }) => {
    const issue = await jira(`/issue/${id}`);
    return { content: [{ type: "text", text: JSON.stringify(issue.fields, null, 2) }] };
  }
);

// Write: move ticket to a new status
server.tool(
  "transition-ticket",
  "Move a Jira ticket to a new status, e.g. from 'In Progress' to 'In Review'.",
  {
    id: z.string().describe("Ticket ID"),
    status: z.enum(["In Progress", "In Review", "Done"]),
  },
  async ({ id, status }) => {
    const transitions = await jira(`/issue/${id}/transitions`);
    const target = transitions.transitions.find((t: any) => t.name === status);
    if (!target) throw new Error(`Transition '${status}' not available for ${id}`);
    await jira(`/issue/${id}/transitions`, "POST", { transition: { id: target.id } });
    return { content: [{ type: "text", text: `${id} moved to "${status}"` }] };
  }
);

// Write: post a comment
server.tool(
  "comment-on-ticket",
  "Add a comment to a Jira ticket. Use to report progress, link PRs, or note blockers.",
  {
    id: z.string().describe("Ticket ID"),
    comment: z.string().describe("Comment text in plain text"),
  },
  async ({ id, comment }) => {
    await jira(`/issue/${id}/comment`, "POST", {
      body: { type: "doc", version: 1, content: [{ type: "paragraph", content: [{ type: "text", text: comment }] }] },
    });
    return { content: [{ type: "text", text: `Comment added to ${id}` }] };
  }
);

The Complete /from-ticket Flow — Now Real

With these tools wired up, the /from-ticket skill from module 5 of Claude for Software Engineers becomes a fully automated pipeline:

text 
1. get-ticket ENG-4521        → reads description, acceptance criteria
2. Explore codebase           → finds relevant files
3. Implement + write tests    → follows project conventions
4. /ship                      → review, lint, commit, push, open PR
5. transition-ticket ENG-4521 "In Review"  → moves ticket
6. comment-on-ticket ENG-4521  → posts PR link and summary

A single command. No human involvement once it starts.

Deploying for Your Team

Two options depending on your team's needs:

Option 1: npm Package (Recommended for Internal Tools)

bash 
# Publish to your internal npm registry
npm publish --registry https://npm.yourcompany.com

# Teammates add it with:
claude mcp add jira -e JIRA_TOKEN=$JIRA_TOKEN -- npx -y @yourcompany/jira-mcp

Option 2: Commit to Repo + .mcp.json

bash 
# Commit your server to the repo
git add .claude/mcp/ && git commit -m "Add internal MCP server"

# .mcp.json points to the local server
{
  "mcpServers": {
    "jira": {
      "command": "node",
      "args": [".claude/mcp/jira/index.js"],
      "env": { "JIRA_TOKEN": "${JIRA_TOKEN}" }
    }
  }
}
💡
Add Your Server to the .claude/ Directory
Storing your MCP server under .claude/mcp/ keeps all Claude Code tooling in one place — alongside skills, agents, hooks, and settings. When a developer clones the repo, they get the full automation suite including your custom MCP server.
🎯
Final Checkpoint
Your server should now connect to at least one private system and have at least one write-back tool. Test the full loop: ask Claude to read a record, make a change in your codebase based on it, then write back to the system (update a status, post a comment). Commit the server and .mcp.json to your repo.
