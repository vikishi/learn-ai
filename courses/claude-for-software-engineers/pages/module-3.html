Module 3

Subagents — Parallel AI Workers

Spawn specialized AI workers with isolated context windows to handle complex tasks in parallel without degrading your main session.

3 lessons · ~85 minutes · Intermediate

Lesson 3-1: Understanding Subagents

25 min · Concept

As your main Claude Code session accumulates context — file contents, conversation history, tool outputs — it approaches the context window limit. Quality degrades. Subagents solve this by spawning fresh AI workers, each with its own isolated context window dedicated to a specific task.

Built-in Subagents

  • Explore — Codebase research and navigation
  • Plan — Architecture design and implementation planning
  • General-purpose — Full-capability worker for complex tasks

Architecture

text 
Primary Agent (your main Claude Code session)
│
├── Subagent A (Explore: codebase research)
│   └── Returns findings to parent
├── Subagent B (general-purpose: run tests in parallel)
│   └── Returns test results
├── Subagent C (Plan: design architecture)
│   └── Returns implementation plan
└── Primary agent synthesizes results & makes edits

Skills vs Subagents

These two systems are complementary, not competing:

  • Skills = reusable instructions (what to do) — they define the task template
  • Subagents = specialized workers (who does it) — they execute with dedicated resources

A skill tells Claude what to do. A subagent is a dedicated worker who does it with its own context window, model, and tool access.

⚠️
Token Cost Awareness
Subagent-heavy workflows can use ~7x more tokens than single-agent approaches. Each subagent gets its own context window, which means duplicate system prompts and project context. Use subagents when the quality benefit justifies the cost — complex tasks, parallel execution, or when your main context is nearly full.

Lesson 3-2: Building Custom Subagents

30 min · Hands-on

Custom subagents live in .claude/agents/ as markdown files with YAML frontmatter specifying their model, available tools, and isolation level.

1. Code Reviewer

markdown 
---
model: sonnet
tools:
  - Read
  - Grep
  - Glob
  - Bash
---
# Code Reviewer Agent

You are a senior code reviewer. Your job is to review changes
for correctness, security, performance, and maintainability.

## Process
1. Read the diff or changed files provided
2. Understand the intent of the changes
3. Check for bugs, security issues, and performance problems
4. Verify tests exist and cover new behavior
5. Provide actionable feedback with severity levels

## Output Format
- 🔴 **Critical**: Must fix before merge
- 🟡 **Warning**: Should fix, not blocking
- 🟢 **Suggestion**: Nice to have improvement
- ✅ **Approved**: No issues found

Be specific. Reference line numbers. Suggest fixes.

2. Security Auditor

markdown 
---
model: opus
tools:
  - Read
  - Grep
  - Glob
  - Bash
---
# Security Auditor Agent

You are a security specialist focused on finding vulnerabilities
in application code. Think like an attacker.

## Audit Checklist
1. **Injection** — SQL, NoSQL, command, LDAP injection vectors
2. **Authentication** — weak passwords, missing MFA, session issues
3. **Authorization** — privilege escalation, IDOR, missing checks
4. **Data Exposure** — secrets in code, PII leaks, verbose errors
5. **Dependencies** — known CVEs, outdated packages, supply chain
6. **Configuration** — debug mode, default credentials, CORS

## Process
1. Scan for hardcoded secrets and credentials
2. Trace user input from entry point to database/system calls
3. Check authentication and authorization at every boundary
4. Review dependency versions against known CVE databases
5. Verify error handling doesn't expose internals

## Output
Report findings with CVSS-like severity and remediation steps.

3. Test Writer

markdown 
---
model: sonnet
tools:
  - Read
  - Grep
  - Glob
  - Bash
---
# Test Writer Agent

You are a test engineer who writes comprehensive, maintainable tests.

## Principles
- Test behavior, not implementation
- One assertion per test (when practical)
- Descriptive names: "should [behavior] when [condition]"
- Arrange-Act-Assert pattern
- Mock external dependencies, not internal modules

## Process
1. Read the source file to understand the public API
2. Identify existing test patterns in the project
3. Write tests covering:
   - Happy path (expected inputs → expected outputs)
   - Edge cases (empty, null, boundary values)
   - Error cases (invalid inputs, failures)
4. Run the tests to verify they pass
5. Check coverage if tools are available

## Output
Complete test files ready to commit. Follow project conventions.

Key Frontmatter Options

  • model — Which Claude model to use (haiku, sonnet, opus)
  • tools — Which tools the agent can access
  • isolation — Context isolation level from parent
✅
Checkpoint
Create all three subagents in your .claude/agents/ directory. Test each one by invoking them with real code changes. Verify they produce useful output within their specialized domain.

Lesson 3-3: Orchestrating Agent Teams

30 min · Hands-on

Real power comes from orchestrating multiple subagents as a team. Each agent handles its specialty while the primary agent coordinates the workflow.

Full Agent Team Structure

text 
.claude/agents/
  explorer.md        # Codebase research & navigation
  architect.md       # System design & planning
  implementer.md     # Code writing & refactoring
  test-writer.md     # Test generation & coverage
  code-reviewer.md   # Quality & correctness review
  security-auditor.md # Security vulnerability scanning

Build Validator (Lightweight Agent)

markdown 
---
model: haiku
tools:
  - Bash
---
# Build Validator Agent

You are a build validation agent. Your only job is to run
build commands and report pass/fail status.

## Process
1. Run the build command: `npm run build`
2. Run the linter: `npm run lint`
3. Run the test suite: `npm test`
4. Report results in this format:

## Results
- Build: ✅ PASS / ❌ FAIL (error summary)
- Lint: ✅ PASS / ❌ FAIL (violation count)
- Tests: ✅ PASS (X/Y) / ❌ FAIL (failures listed)

Keep output concise. Only include error details for failures.

Orchestration Flow

The optimal orchestration pattern runs agents in stages, parallelizing where possible:

  1. Explorer — Research the codebase area (sequential, feeds into planning)
  2. Architect — Design the solution based on exploration findings
  3. Implementer + Test Writer — Execute in parallel (independent tasks)
  4. Code Reviewer + Security Auditor — Review in parallel (independent checks)
  5. Primary Agent — Synthesize feedback, make final adjustments, commit
text 
Stage 1 (Sequential):   Explorer → findings
Stage 2 (Sequential):   Architect → plan (uses findings)
Stage 3 (Parallel):     Implementer → code    |  Test Writer → tests
Stage 4 (Parallel):     Code Reviewer → feedback  |  Security Auditor → report
Stage 5 (Sequential):   Primary Agent → synthesize, fix, commit
💡
Model Selection Strategy
Use Haiku for simple command-running tasks (build validation, formatting checks) where speed and cost matter more than reasoning. Use Sonnet for code generation and review where good reasoning is needed at reasonable cost. Use Opus for deep reasoning tasks (security audits, architecture decisions) where quality is paramount.
