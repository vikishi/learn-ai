Module 2

Skills — Reusable Automation

Build reusable prompt templates that auto-trigger based on context, inject live data, and chain into powerful workflows.

3 lessons · ~80 minutes · Intermediate

Lesson 2-1: Creating Your First Skill

25 min · Hands-on

A skill is a SKILL.md file with YAML frontmatter and markdown instructions. Skills live in a structured directory and auto-invoke based on their description.

File Structure

text 
.claude/skills/
  summarize-changes/
    SKILL.md
  generate-tests/
    SKILL.md
    templates/
      jest-template.md

Example: Summarize Changes Skill

markdown 
---
description: Summarizes uncommitted changes and flags risks.
  Use when the user asks what changed, wants a commit message,
  or asks to review their diff.
---
## Current changes
!`git diff HEAD`

## Instructions
Summarize the changes above in 2-3 bullet points, then list
any risks: missing error handling, hardcoded values, or tests
that need updating.

Key Concepts

  • !`command` = dynamic context injection (runs at invocation time)
  • description in frontmatter = when to auto-invoke the skill
  • Directory name = /slash-command (e.g., /summarize-changes)
  • Personal skills: ~/.claude/skills/ (available in all projects)
  • Project skills: .claude/skills/ (shared with team via git)
💡
Dynamic Context Injection
The !`command` syntax is what makes skills powerful. It runs the command at invocation time and injects the output directly into the prompt. Use it for git diffs, file contents, test results, or any live data your skill needs.

Lesson 2-2: Production Skills — Code Review, Testing, Commits

30 min · Hands-on

Let's build three production-ready skills that you'll use daily.

1. Auto Code Review

markdown 
---
description: Reviews staged changes for bugs, security issues,
  and style violations. Use when the user asks for a code review
  or before committing changes.
---
## Staged Changes
!`git diff --cached`

## Project Standards
!`cat CLAUDE.md`

## Review Checklist
Review the staged changes against project standards. Check for:
1. **Bugs** — null checks, off-by-one errors, race conditions
2. **Security** — injection vulnerabilities, exposed secrets, auth gaps
3. **Performance** — N+1 queries, unnecessary allocations, missing indexes
4. **Style** — naming conventions, file organization, documentation
5. **Tests** — are new paths covered? Are edge cases tested?

Output format:
- 🔴 Critical (must fix before merge)
- 🟡 Warning (should fix, not blocking)
- 🟢 Suggestion (nice to have)
- ✅ Looks good (no issues found)

2. Generate Tests

markdown 
---
description: Generates unit tests for recently changed files.
  Use when the user asks to write tests, add coverage, or after
  implementing a new feature.
---
## Changed Files
!`git diff --name-only HEAD~1`

## Existing Test Patterns
!`find . -name "*.test.*" | head -5 | xargs head -30`

## Instructions
For each changed source file that lacks test coverage:
1. Identify the public API (exported functions/classes)
2. Write tests covering happy path, edge cases, and error cases
3. Follow the existing test patterns shown above
4. Place test files next to source files
5. Use descriptive test names: "should [expected behavior] when [condition]"

3. Commit Message Generator

markdown 
---
description: Generates a conventional commit message from staged changes.
  Use when the user asks for a commit message or is ready to commit.
---
## Staged Changes
!`git diff --cached --stat`

## Detailed Diff
!`git diff --cached`

## Instructions
Generate a commit message following Conventional Commits:
- type(scope): subject (max 50 chars)
- blank line
- body: what changed and why (wrap at 72 chars)
- blank line
- footer: breaking changes, issue refs

Types: feat, fix, refactor, docs, test, chore, perf, ci
Scope: the module or area affected
✅
Checkpoint
Create all three skills in your .claude/skills/ directory and test them with real code changes. Make a small edit, stage it, and invoke each skill to verify they work correctly.

Lesson 2-3: Chaining Skills into Workflows

25 min · Hands-on

Individual skills are powerful, but chaining them into workflows creates end-to-end automation. A workflow skill orchestrates multiple steps in sequence.

Full Feature Workflow

markdown 
---
description: Implements a complete feature from exploration to commit.
  Use when the user describes a feature to build or a task to complete.
---
## Project Context
!`cat CLAUDE.md`

## Current Branch
!`git branch --show-current`

## Workflow Steps

### Step 1: Explore
Understand the codebase area relevant to this feature.
Read related files, understand existing patterns.

### Step 2: Plan
Create a brief implementation plan:
- Files to create/modify
- Key design decisions
- Edge cases to handle

### Step 3: Implement
Write the code following project conventions from CLAUDE.md.
Keep changes minimal and focused.

### Step 4: Test
Write tests for the new code. Run existing tests to check
for regressions: !`npm test 2>&1 | tail -20`

### Step 5: Review
Review your own changes against the project checklist.
Use /auto-code-review if available.

### Step 6: Commit
Generate a conventional commit message and commit the changes.
Use /commit-message if available.

Weekly Code Review Report

markdown 
---
description: Generates a weekly summary of code changes, patterns,
  and technical debt. Use on Fridays or when the user asks for a
  weekly report.
---
## This Week's Commits
!`git log --oneline --since="7 days ago"`

## Files Changed This Week
!`git diff --stat HEAD~$(git rev-list --count --since="7 days ago" HEAD)`

## Instructions
Generate a weekly report with:
1. **Summary** — Key features shipped, bugs fixed
2. **Patterns** — Recurring themes in changes
3. **Technical Debt** — Areas that need refactoring
4. **Recommendations** — What to focus on next week

Key Insight

Skills can reference other skills and subagents by name. This creates a layered architecture where simple skills compose into complex workflows, and workflows can delegate specialized tasks to subagents.

💡
The Orchestration Pattern
Think of workflow skills as orchestrators. They don't do the deep work themselves — they sequence steps, pass context between them, and delegate specialized tasks. This keeps each piece simple and testable while the workflow handles coordination.
