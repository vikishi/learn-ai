Module 8

Documentation Automation

Build living documentation that auto-updates when code changes — API docs, changelogs, and onboarding guides that never go stale.

1 lesson · ~25 minutes · Advanced

Lesson 8-1: Living Documentation Skills

25 min · Hands-on

Documentation that auto-updates when code changes. These three skills ensure your docs are always current — no more "the docs are outdated" excuses.

1. /doc-api — Generate API Documentation

Automatically generates OpenAPI-compatible documentation from your route handlers:

markdown 
---
description: Generates API documentation from route handlers.
  Auto-invokes when the user creates or modifies API routes.
---
## Current routes
!`grep -rn "router\." src/routes/ | head -50`

## Instructions
1. Read all route handler files
2. For each endpoint, document:
   - Method and path
   - Request body schema (from Zod/Joi validators)
   - Response schema with examples
   - Authentication requirements
   - Error responses
3. Generate OpenAPI-compatible documentation
4. Save to docs/api/

2. /changelog — Generate Changelog from Commits

Transforms raw git history into a user-friendly changelog, grouped by category:

markdown 
---
description: Generates a user-friendly changelog from git history.
disable-model-invocation: true
---
## Recent commits
!`git log --oneline --since="1 week ago"`

## Instructions
Group commits into:
- ✨ Features — New functionality
- 🐛 Bug Fixes — Corrections
- ⚡ Performance — Speed improvements
- 🔒 Security — Vulnerability patches
- 📖 Documentation — Doc updates
- 🔧 Maintenance — Refactoring, dependency updates

Write in user-friendly language. Append to CHANGELOG.md.

3. /onboard — Generate Onboarding Guide

Creates a comprehensive onboarding document for new team members by analyzing the actual project:

markdown 
---
description: Generates an onboarding guide for new developers.
  Use when a new team member joins or project setup changes.
---
## Project info
!`cat package.json | head -20`
!`cat README.md | head -50`

## Instructions
Generate a comprehensive onboarding guide covering:

1. **Prerequisites** — Required tools, versions, accounts
2. **Environment Setup** — Step-by-step local dev setup
3. **Project Structure** — Directory layout with explanations
4. **Development Workflow** — Branch strategy, PR process
5. **How to Run** — Dev server, build, watch mode
6. **How to Test** — Unit, integration, e2e commands
7. **How to Deploy** — Staging and production processes
8. **Architecture Decisions** — Link to ADRs or summarize key choices
9. **Common Gotchas** — Things that trip up new developers
10. **Key Contacts** — Who to ask about what

Save to docs/ONBOARDING.md
💡
Auto-Update Docs on Route Changes
Add a PostToolUse hook that reminds Claude to update docs when route files change. This ensures API documentation stays in sync with the actual implementation.
🎯
Checkpoint
Run /doc-api and /changelog on your project — would these help your team? Customize the templates to match your project's conventions.
