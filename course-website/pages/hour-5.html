Part 1: Pre-Commit Review Flow (20 min)

Daily Development Flow

text
1. Write code
   ↓
2. "Review auth.py" (code-reviewer)
   ↓
3. Fix issues found
   ↓
4. "Pre-commit check auth.py" (quick validation)
   ↓
5. Commit if approved
   ↓
6. Continue coding

Quick Commands

CommandWhat It Does
"Pre-commit check [file]"Security (critical only) + bugs + test verification → Go/No-Go
"Full review of [file]"Complete security + code review + tests + refactoring
"Quick security and bug check"Fast scan, skip refactoring and style
"Is this ready for PR?"Security + review + verify tests pass + team standards check

Before Creating a PR

text
1. "Full review on all changed files"
   ↓
2. Address critical issues
   ↓
3. "Generate tests for uncovered code"
   ↓
4. Run full test suite
   ↓
5. Create PR with confidence

Part 2: Repository-Wide Analysis (20 min)

Create .claude/agents/codebase-auditor.md:

markdown
---
name: codebase-auditor
description: Audits entire repositories for hotspots, test coverage gaps, dependency vulnerabilities, and technical debt
tools: Read, Bash, Grep, Glob
---

# Codebase Auditor Subagent

You analyze entire repositories for patterns, issues, and technical debt.

## Analysis Steps

### 1. Repository Overview
- Total files, lines of code, commits
- Top contributors, repository age

### 2. Identify Hotspots
- Most changed files (high churn = likely bugs)
- Largest files (may need splitting)
- Complex files (deep nesting, many branches)

### 3. Test Coverage
- Overall percentage vs. target
- Untested critical paths
- Missing edge case coverage

### 4. Dependency Health
- Outdated packages
- Known security vulnerabilities (pip-audit, npm audit)
- Deprecated dependencies

### 5. Code Quality Patterns
- TODOs/FIXMEs count
- Hardcoded values
- Large files (>500 lines)
- Duplicated code patterns

### 6. Output Report
```
# Codebase Audit Report

## Overview (stats)
## Hotspots (high-churn files)
## Test Coverage (gaps)
## Dependencies (vulnerabilities)
## Technical Debt (TODOs, large files)
## Security Concerns
## Recommendations
  - Immediate (this week)
  - Short Term (this month)
  - Long Term (this quarter)
## Risk Assessment
```

Run this audit monthly to track improvements.

Part 3: Domain-Specific Reviewers (20 min)

Build reviewers for your specific work domain. Example — API Reviewer:

markdown
---
name: api-reviewer
description: Reviews REST API code for correct HTTP status codes, input validation, auth checks, and consistent response format
tools: Read, Bash, Grep, Glob
---

# API Development Reviewer

## Checks
- Correct HTTP status codes (200, 201, 400, 401, 403, 404, 500)
- Input validation on all endpoints
- Authentication on all non-public endpoints
- Authorization (user owns the resource?)
- Consistent response format
- Rate limiting on write endpoints
- API versioning
- Documentation (OpenAPI/Swagger)

Other domain reviewers you might build:

  • Frontend Reviewer — React hooks, accessibility, performance
  • Database Reviewer — Query optimization, migrations, indexes
  • DevOps Reviewer — Dockerfile, CI/CD pipelines, infrastructure
💡
Build What You Need

The most valuable reviewer is one tailored to YOUR biggest pain points. What type of bugs do you ship most often?
