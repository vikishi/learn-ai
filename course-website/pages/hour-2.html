Building a Code Reviewer Subagent

Create a comprehensive code reviewer that provides specific, actionable feedback on code quality.

60 min
Hands-on

Part 1: What Makes a Good Code Review? (10 min)

Good Reviews:

  • ✅ Catch real issues before they reach production
  • ✅ Provide specific, actionable feedback
  • ✅ Explain WHY something is a problem
  • ✅ Suggest concrete fixes
  • ✅ Stay quiet when code is good

Bad Reviews:

  • ❌ Nitpick style without explaining impact
  • ❌ Give vague advice ("consider improving this")
  • ❌ Overwhelm with minor issues
  • ❌ Miss critical bugs while focusing on formatting

Review Priority Categories

PriorityCategoryExamples
1CorrectnessLogic errors, data corruption, wrong assumptions
2SecurityInjection, auth issues, data exposure
3PerformanceN+1 queries, inefficient algorithms, memory leaks
4MaintainabilityCode clarity, error handling, test coverage

Part 2: Writing Review Instructions (20 min)

Create .claude/agents/code-reviewer.md:

markdown
---
name: code-reviewer
description: Reviews code for bugs, logic errors, security issues, performance problems, and maintainability
tools: Read, Bash, Grep, Glob
---

# Code Reviewer Subagent

You are an experienced software engineer reviewing code for quality and correctness.

## Review Process

### 1. Understand Context First
- Read the file and understand what it does
- Check related files if needed (imports, dependencies)
- Review recent changes: `git log -10 --oneline [filepath]`
- Look at git diff if changes are uncommitted

### 2. Analysis Focus Areas

#### CRITICAL (Always Check)
- Logic errors, incorrect conditionals, off-by-one
- Security vulnerabilities (injection, auth bypass, secrets)
- Data loss risks, missing transactions
- Unhandled errors, null/undefined access

#### IMPORTANT (Usually Check)
- Performance (N+1 queries, O(n²) algorithms)
- Race conditions, shared state in async code
- Missing test coverage for critical paths

#### MINOR (Mention If Obvious)
- Code clarity, poor naming
- Long functions (>50 lines)
- Not following language idioms

### 3. Output Format

[SEVERITY] Line X-Y: [Issue Type]

Issue: [Clear description]
Impact: [Why this matters]
Fix: [Specific solution with code]

Severity: CRITICAL | HIGH | MEDIUM | LOW

### 4. Review Principles
- Be Specific — cite exact line numbers and code
- Explain WHY — don't just say "this is wrong"
- Be Actionable — provide concrete fixes
- Stay Focused — if code is clean, say so
- Prioritize — critical bugs over minor style

### 5. What NOT to Report
- Style preferences (unless project has standards)
- Performance optimizations without proof
- Theoretical vulnerabilities not exploitable
- Issues already caught by existing tests

### 6. When Code Is Clean

If no significant issues found:
```
Code review complete. No issues found.

Positive observations:
- ✓ Proper error handling
- ✓ Good test coverage
- ✓ Clear, readable code

This code is ready to merge.
```

Part 3: Testing & Refinement (30 min)

Test your reviewer against four scenarios:

Test 1: Logic Bug

python
def calculate_discount(price, discount_percent):
    # BUG: Should divide discount_percent by 100
    return price - (price * discount_percent)

print(calculate_discount(100, 10))  # Should be 90, will be -900

Test 2: Security Vulnerability

python
import sqlite3

def get_user_by_name(username):
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    # SQL injection vulnerability!
    query = f"SELECT * FROM users WHERE username = '{username}'"
    cursor.execute(query)
    return cursor.fetchone()

Test 3: Clean Code (No Issues)

python
from typing import List, Optional
from decimal import Decimal

def calculate_total(items: List['Item']) -> Decimal:
    if not items:
        return Decimal('0.00')
    try:
        total = sum(Decimal(str(item.price)) for item in items)
        return total.quantize(Decimal('0.01'))
    except (ValueError, AttributeError) as e:
        raise ValueError(f"Invalid item in list: {e}")
⚠️
Key Test: Clean Code

A good reviewer should acknowledge clean code without forcing false issues. If your agent nitpicks good code, tighten the instructions.
